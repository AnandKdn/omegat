文件筛选器

警告

文件筛选器要么是局部且专属于给定项目的，要么是全局且可用于共享配置文件夹的所有项目的。

详情请参阅：

  • Local File Filters...

  • 全局文件筛选器

  • 配置文件夹

以粗体显示的筛选器是在当前项目中所使用的。

如果不想翻译某个筛选器所关联的文件，可以通过取消对其勾选框的勾选来禁用它。这些文件的内容就不会显示供翻译。

注意

可以根据名称或者是否启用来对筛选器进行排序。点击相关表头即可按升序或降序对它们进行排序。

要修改某个筛选器所关联的文件扩展名、译文文件名和编码，请在列表中选择它并单击“编辑...”按钮。

一些筛选器提供了“选项...”按钮来进一步自定义其设置。

点击“恢复默认值”按钮可将文件筛选器重置为其默认设置。

修改过的全局文件筛选器首选项保存在配置文件夹的 filters.xml 中。详情请参阅配置文件夹。删除此文件也会重置筛选器首选项。

修改过的局部文件筛选器首选项保存在位于项目文件夹内的 filters.xml 文件中。详情请参阅章节Project Folder。删除此文件也会重置筛选器首选项并将项目恢复为使用全局文件筛选器。

公共首选项

隐藏开头和末尾的标签

OmegaT 通常都需要开头和末尾的标签才能正确地重新创建译文片段。将其从可翻译内容中隐藏掉可确保你不会错误地删除或修改它们。

如果你保留了开头和末尾处的标签，请确保译文中也包含它们。

删除非分割项目中开头和末尾的空白字符

默认情况下，OmegaT 会从可翻译内容中移除开头和末尾的所有空白字符。在未分割的项目中，禁用此选项可以让开头和末尾的空白字符在译文中可修改。

为所有标签保留空格

如果原文文档中包含用于控制排版布局的空白字符，则这些空白字符在译文文档中也必须保留。

不要用文件名来标识多重译文

原文文件名是表征备选译文的元素之一。如果勾选此选项，则只使用前后片段或片段标识符来表征备选翻译。

在其他文件中具备相同表征的片段将以相同方式翻译。

编辑

双击可编辑字段来进行简单编辑，或者点击“编辑...”按钮来访问修改对话框。

要添加筛选器模式，点击“添加...”来打开一个类似的对话框。

以上两种对话框都允许你自定义此筛选器所关联的原文和译文文件名模式并可以选择各自的编码。

用“文件名变量”下拉菜单来自定义译文文件名。

原文文件名模式

为了将筛选器和文件关联起来，OmegaT 会检查其文件扩展名并尝试与筛选器中的原文文件名模式进行匹配。

例如，XHTML 筛选器中注册的模式 .xhtml 会匹配任何具有 xhtml 扩展名的文件。如果在 source 文件夹中找到这样的文件，那么此文件就会由 XHTML 筛选器处理。

你可以修改或添加文件名模式来将不同的文件关联给筛选器。

警告

将文件扩展名关联给筛选器并不足以让筛选器正确处理该文件。文件结构还必须与此筛选器兼容。即便将 .odt 关联给 XHMTL 筛选器，这个筛选器也无法理解 LibreOffice Writer 文件的内容。

原文文件名模式使用通配符：* 字符匹配零个或更多个字符，而 ? 字符只匹配一个字符。

例如，如果要让文本筛选器处理 readme 文件（readme、read.me、或readme.txt），使用模式 read*。

原文文件编码和译文文件编码

大多数文件格式允许各种可能的编码。默认情况下，译文文件的编码和原文文件相同。

原文文件编码和译文文件编码字段用下拉菜单列出了所有支持的编码。选择“<自动>”选项让 OmegaT 基于以下条件自行选择编码：

  • 对于基于 XHTML 或 XML 的文件，如果原文文件中存在编码声明，那么 OmegaT 会用它来识别编码。

  • 对于某些文件格式（例如 Java properties），OmegaT 会使用某个强制性编码。

  • 对于文本文件，OmegaT 使用操作系统的默认编码。

分割规则

Paragraph or sentence?

Translation memory tools work with textual units called segments. When a translation is entered, the segment containing the source text is stored with its translation in the project memory, and subsequently used to match other source segments in the project.

Use the Sentence-level segmenting project property to select the type of segmentation.

Segments are by default paragraphs defined by the file format itself.

Not using sentence segmentation on a document is equivalent to using paragraph segmentation. In that case, each paragraph (as defined in the original document format) is displayed as a single segment, and the translator is free to reorganize the sentences within the segment in the translation.

Paragraph segmentation works well with more literary or creative texts, as well as, more generally, with documents for which translation memory matches are not so important.

Sentence segmentation relies on a number of rules (called segmentation rules ) that define what constitutes a sentence in the source language. This setting works well with documents where repetitions or similar sentences are common, such as technical or legal documents.

Paragraph-Level segmentation

OmegaT first parses the text for paragraph-level segmentation. This process relies only on the structure of the source file to produce segments.

For example, text files may be segmented on line breaks, empty lines, or not at all. Files containing formatting (ODF, HTML, or other documents) are divided at block-level (paragraph) tags. Translatable object attributes in XHTML or HTML files can be extracted as separate "paragraphs".

Sentence-level segmentation

After dividing the source file into structural units, OmegaT further divides those units into segments.

You can visualize segmentation as the process of moving the cursor along the text, one character at a time, and looking for the position where a break will occur, or where a break will not be allowed.

Each time the cursor moves to the next character, OmegaT checks whether:

  • the text before the location corresponds to a Before rule,

  • and the text after the location corresponds to the associated After rule.

If the location matches both rules, it is considered either as a break, or as a non-break, depending on what the rule defined.

Global or local?

注意

The same mechanisms and dialogs are used to define global and local segmentation rules.

By default, segmentation settings are global and shared by all projects.

Use the Local Segmentation Rules... project property to limit the scope of the segmentation rules to the current project.

You can achieve a similar result by starting OmegaT from the command line. See the Command line launch how-to for details.

If you use local rules, you can still access the global rules, but modifying them will have no effect on your project.

规则

OmegaT provides predefined segmentation rules, and the translator can use regular expressions to modify them. See the 正则表达式 appendix for details.

As a reminder, rules work the following way: when a rule matches, OmegaT puts a marker at the match location so that rules that come after ignore that location. That is the reason why exception rules must come before segmentation rules.

警告

If you change the segmentation while translating, you will have to reload the project for the new segmentation to take effect. This will split or merge some previously translated segments, which will therefore no longer be considered translated. Nonetheless, their original translation will still be in the project memory.

这里有一些例子。

分割规则

分割规则和语言模式是用正则表达式定义的。你可以随意修改它们以改进文档的片段分割或者添加其他通用规则。详情请参阅附录分割规则。

分割规则或例外规则定义了在片段中进行拆分或不能拆分的位置。定义该位置需要两个正则表达式：“前”表达式定义了需要应用该规则的位置之前的文本模式，而“后”表达式这定义此位置之后的文本模式。

与项目的原文语言相匹配的语言模式将应用于该项目。

4 条规则

正则表达式用于查找文本，包括在屏幕上或打印输出时不可见的字符，例如空格、tab、或断行。任何给定表达式要么 匹配 要么 不 匹配 单词、片语、或其他文本序列。

在确定匹配时，表达式中的每一个字符都是有实际意义的。

许多字符或字符组合在正则表达式中具有特殊含义。

警告

正则表达式只匹配文本。它们不会匹配诸如 粗体 、 斜体 等修饰或其他 格式效果 。

有四条规则需要牢记。

大多数字符只是匹配其自身

正则表达式中的大多数字符只是在文本序列中 查找其自身 。

例如，以七个字母拼出的单词“ example ”只是告诉搜索功能要按此顺序精确匹配这些字母。简而言之，这样的搜索只是在查找单词“ example ”。

前面有反斜杠（\）的字母具有特殊含义

与单独的字母（如上文所述，仅表示其自身）不同，前面有 \ 的字母在正则表达式中具有特殊功能。

例如， r 只是一个普通的字符，但前面加上 \ 变成 \r 就将其变为匹配 回车字符 的特殊组合。类似地，\R 匹配 任意换行符 。

注意

字母中只有 i j l m o 和 y （大写和小写都算）在其前面加上反斜杠时没有特殊含义。本手册只介绍一小部分具有特殊含义的字母。

关于这里未涵盖的组合的信息，请参阅下文References部分中的站点。

默认情况下，有十二个字符具有特殊含义

这种特殊含义必须用另一个字符来取消才能匹配此字符本身。

完整的字符列表呈现在下文。范例之一是 .：单独出现时，它具有匹配 任意单个字符 的特殊含义。

要查找普通的句点，就必须用 \ 来取消此含义，这就形成了表达式 \.，它正好匹配一个句点。

\ 字符是个非常特殊的字符

如上所述，\ 字符具有取消或激活其他字符的特殊含义的默认特殊含义。如果将其放在没有特殊含义的字符（无论是默认还是附加）之前，它都不会有任何效果。

可以通过双写 \ 来取消其自身的特殊含义，即 \\ 的形式，这只会匹配 反斜杠 字符本身。

12 个字符

十二个特殊字符是 反斜杠 \、 脱字符 ^、 美元符号 $、 句点 （或者叫 点号 ）.、 竖线 （或者叫 管道符号 ）|、 问号 ?、 星号 *、 加号 +、左 括号 (、右 括号 )、左 方括号 [、和左 花括号 {。

下面用依赖于各个字符的正则表达式以及与其匹配或不匹配的文本来作为例子简要介绍每个字符。

反斜杠：\

此字符取消或激活后继字符的特殊含义。

  0\.[0-9]
匹配

0.0 到 0.9 的数字，或者诸如 10.5 或 560.5 之类的数字中最后的 0.5 。

\. 取消了句点的“任意字符”的含义，让其匹配小数点，而 \d 将普通的小写“d”字母转变为匹配任意 0 到 9 数字的表达式。
不匹配

诸如 0、1、0-3 这样的序列，或 0x002E 的前三个字符（如果正则表达式只是 0.[0-9] 就能匹配了，其中句点前面没有反斜杠）
脱字符：^

当它是表达式的第一个字符时， 脱字符 这个字符匹配一行的开头。

当它是方括号括起的字符类中的第一个字符时，它匹配不属于该类的所有字符。

 

  1. ^A

  2. [^abc]
匹配

  1. The uppercase “A” 以下句子中的大写“A”：“A long, but exciting journey was about to begin”。

  2. 任意 不是 “a”、“b”、或“c”的字符。例如，在单词“back”中，只有“k”是匹配的。
不匹配

  1. 以下句子中的大写“A”：“My friend is writing a book called A Long Journey ”。

  2. 单词“back”中的小写“a”、“b”、或“c”。
美元符号：$

当它是表达式的最后一个字符时， 美元 符号匹配一行的末尾。

  ^\w+:$
匹配

由单个单词组成并以冒号结尾的行：

不匹配

由单个单词组成但不以冒号结尾的行：

句点：.

匹配任何单个字符。

  c.t
匹配

任何以“c”开头并以“t”结尾的三字母组合：“ cat ”、“ cut ”、“ cot ”、或者甚至诸如“ czt ”或“ cqt ”这样无意义的组合。
不匹配

包含以“c”开头并以“t”结尾、但拆分成多行的三字母组合：

竖线：|

此字符起到了“或”的作用，匹配其前面或者后面表达式任一者。

  ^An|^The
匹配

诸如以下片语中开头的“An”或“The”，比如：

“An apple a day…”
“The apple of my eye…”

不匹配

诸如以下片语中开头的“An”或“The”，比如：

“A story called An Unsung Hero .”
“They work for The Daily Post .”

问号：?

此字符指定应该匹配前面那个字符或表达式的零个或一个实例。

  an?␣（其中的“␣”代表单个空格）
匹配

下面其中的“a ”或“an ”：

“I have a question.”
“I know an excellent doctor.”

它也会找到诸如“Can I help you?”这样的句子中“Can ”最后的“an ”，或者“We had pasta for lunch.”中“pasta ”最后的“a ”。
不匹配

下面其中的“a”或“an”：

The indefinite article: “a” (or “an”).

它们后面没有跟着空格。
星号：*

此字符指定应该匹配前面那个字符或表达式的零个或更多个实例。

  run\w*
匹配

单词“run”，还有“runs”、“runner”、“runway”、“grunt”或“brunt”中的“runt”，以及包含“run”后面跟着零个或更多个“ 单词字符 ”（包括数字和下划线，因此诸如 run_123@example.email.org 这样的 Email 地址中“@”之前的部分也是匹配项）的任意其他单词或字符序列。
不匹配

整个“run-on”或“run'n'gun”，因为 \w 中不包括连字符和撇号。这些片语中只有开头的“run”才是匹配的。
加号：+

此字符指定应该匹配前面那个字符或表达式的一个或多个实例。

  \d+.d
匹配

诸如“1.5”、“23.2”或“5235.8”这样小数点前面有任意多位但小数点后只有一位的数字。
不匹配

整个“5,235.8”或“21,571.9”。只有千位分隔符后面的部分会被匹配。
左括号：(

此字符开启一个 分组 ，这是一组被视为单一单元的字符。分组会被编号，其内容组存储在内存中。之后可以在搜索表达式中用 \n 来重用它们，其中 n 是分组的编号。

注意

分组的内容可以用在替换文本中。使用 $n ，其中 n 是在搜索中该定义的分组编号。

括号始终要左右成对使用，试图单独使用左括号或右括号会导致错误。

  (\b\w+\b)\h\1\b
匹配

空格隔开的重复单词，例如以下句子中的连续“an”：

“I bought an an apple.”
不匹配

以下句子中的“that, that”：

“But that, that is just unbelievable”，因为第一个“that”后面跟的是逗号加上空格而非仅只一个空格。
右括号：)

此字符结束一个分组。它十分特别，因为它永远不能单独使用。如果需要匹配右括号字符本身，则必须在其前面加上 \。

  ^\d+\)
匹配

列表中每一行开头的序号（包括括弧），诸如：

1) Apples
2) Oranges
3) Pears
不匹配

不在行首处的序列号。

Follow these steps:

Step 1) Preparation
Step 2) …
左方括号：[

此字符必须与右方括号成对出现，括起一组单个字符，其中每个字符都代表一个有效的潜在匹配项。

只有左方括号是特殊的，需要在前面加上反斜杠才能搜索方括号字符本身。如果只需要匹配右方括号本身，并不需要在前面加上反斜杠。（你还是可以加，但这并不会对表达式或结果产生任何影响。）

  li[cs]en[cs]e
匹配

正确的“licence”和“license”拼写，以及潜在的错误拼写“lisence”和“lisense”。
不匹配

诸如“licensse”或“lissense”这类更严重的错误拼写。
左花括号：{

此字符必须与右大括号成对出现，括起一个 精确数字 、 最小值 、 最大值 、或 区间 ，用于指定应匹配前一个字符或分组实例的数量。

只有左花括号是特殊的，需要在前面加上反斜杠才能搜索花括号字符本身。如果只需要匹配右花括号本身，并不需要在前面加上反斜杠。（你还是可以加，但这并不会对表达式或结果产生任何影响。）

  \d{4}/\d{1,3}
匹配

诸如“1234/5”、“1472/69”、或“9513/842”这样由四个数字、一个斜杠再加一到三个数字组成的代码。
不匹配

诸如“123/45”、“1472/6985”、或“95133/15746”这样的代码。

注意： 虽然上面的代码中最后两个并不完全匹配，但是表达式会返回“1472/6985”中的“ 1472/698 ”部分，以及“95133/15746”中的“ 5133/157 ”部分。

大小写

默认情况下，常规的 OmegaT 搜索不区分大小写：它们同时匹配大小写字符，除非你启用选项选项。这样做会让整个搜索表达式区分大小写。

相比之下，正则表达式在默认情况下是区分大小写的。举例说，这意味着用正则表达式“OmegaT”进行搜索是不会匹配“omegat”的。但是，正则表达式还提供特殊修饰符来在表达式内指定区分大小写：

(?i)

让修饰符右侧的表达式片段不区分大小写。

(?-i)

让修饰符右侧的表达式片段区分大小写。

可以利用这一点在搜索中对大小写区分进行精细控制。举例说，假设要查找“OmegaT”和“omegat”的实例但不想要“OMEGAT”。可以用以下表达式做到这点：(?i)o (?-i)mega (?i)t，这表示一个不区分大小写的“o”后面跟着区分大小写的“mega”然后在跟上一个不区分大小写的“t”。

类

正则表达式允许你创建字符集合——称为 类 。搜索将匹配集合中的任何一个字符。

类是通过将所需的字符括在方括号中来定义的，并且可以通过罗列每一个需要包括的字符或者通过指定字符区间来指定。例如，可以创建 [£€$] 类来在文本中查找这三个货币符号中的任何一个，或创建 [1-3] 来查找数字 1、2 或 3。

注意

在一个类中，只有反斜杠（\）、脱字符（^）、右方括号（]）和连字符（-）是特殊的。十二个字符的其他几个则是普通的，如果需要搜索这些字符本身，不必在前面加上反斜杠。

可以通过在前面加上反斜杠来搜索这四个类特殊字符本身。对于脱字符、右方括号和连字符，还可以通过将其放在不会触发其特殊含义的位置来搜索其本身：对于脱字符，是除了紧跟在左方括号的位置之外的任何位置；对于右方括号，是紧跟在左方括号之后、或紧跟在左方括号后面的脱字符之后的位置；对于连字符，是紧跟在左方括号之后或紧挨在右方括号之前的位置。

许多常用的集合都有一个反斜杠后跟一个字母组成的简写形式。例如，\d 是 [0-9] 的简写，它匹配 0 到 9 之间的任意数字，后面字母的对应大写用于取反这个类：\D 匹配任何 不是 数字的字符。

More advanced expressions

Some expressions specify a position rather than a character. They indicate where in the text to look for the match, but do not include any characters in that match. The table below list a few of the more common examples. Consult the sites in the References section for more information.

行首
$

行尾
\b

单词边界
\B

Not a word boundary
(?=u)

A character followed by a “u”.

For example, q(?=u) matches the letter “q” when it is followed by a “u”. It therefore matches the “q” in “equal” or “question”, but not the one in “qigong” or “Iraq”.
(?!u)

A character that is not followed by the letter “u”.

For example, q(?!u) matches the letter “q” when it is not followed by a “u”. It therefore matches the “q” in “qigong” or “Iraq”, but not the one in “equal” or “question”.
(?<=q)

(?<!q)

A character that is not preceded by the letter “q”.

For example, (?<!q)u matches the letter “u” if it does not come after “q”. It therefore matches the “u” in “run”, but not the one in “quick”.

词汇表

词汇表是存储在 glossary 文件夹中的术语文件。

片段中所有在 任意 词汇表中存在匹配的术语都会显示在 词汇表 窗格中。

原文术语可以是多个单词的语句。

有 2 种词汇表文件：

项目词汇表

用 编辑 Create Glossary Entry... C + S + G 可在词汇表中输入新词汇。因此，它被称为 可写 词汇表。

用 项目 访问项目内容 可直访问它。然后可以用文本编辑器打开它并进行修改。

并不需要事先准备此文件。

它会在首次往词汇表中添加词条时创建。

注意

如果选择使用一个现有文件作为默认词汇表，则默认情况下，所有新词条会以 tab 分隔的格式进行记录并以 UTF-8 保存。

如果要指定不同的编码，可以通过添加以下格式的“魔术”注释：

# -*- coding: <charset> -*-,

其中 <charset> 通常是 IANA Charset Registry 中所列出的集合之一。

参考词汇表

它们是 OmegaT 可识别格式的术语文件。你不能像项目词汇表那般从 OmegaT 的界面修改它们，但可以在文本编辑器中做到这点。

注意

对任何一个词汇表做出的修改都会立刻被 OmegaT 识别到并显示在 词汇表 窗格中。

词汇表文件夹

默认情况下，每个项目都包含一个 glossary 文件夹，用于存储可写词汇表和需要添加到项目中的所有参考词汇表。详情请参阅 Glossary files folder 项目属性。

所有词汇表必须位于 glossary 项目文件夹中。位于其中嵌套的文件夹下的词汇表也会被识别。

在该参考词汇表文件夹中，你可以创建多个参考词汇表文件夹，按主题、客户、或任何其他适合于工作流程的分类方式来组织。

用 Glossary files folder 项目属性来设置参考词汇表文件夹的位置。此文件夹可以设置为使用在项目之外的文件夹或其他项目的特定子文件夹之一。

项目词汇表

默认情况下，可写项目词汇表位于 glossary 文件夹中，名为 glossary.txt。

可以在 可写词汇表文件 对话框中更改其名称和位置，但必须使用 .txt 或 .utf8 扩展名，并且将其存储在 glossary 文件夹或其子文件夹内。

文件格式

OmegaT 的词汇表文件是包含三列列表的简单纯文本文件，第一列是词汇原文，第二列是词汇译文，而第三列则是注释。第二列和第三列都是可选的。

词汇表可以是“tab 分隔的值”（TSV）或“逗号分隔的值”（CSV）文件，也可以使用 TermBase eXchange（TBX 2）格式。

由 OmegaT 为项目创建的可写词汇表将是以 UTF-8 保存的 TSV 文件。用户创建的仅使用拉丁字符的文件如果不包含非 ASCII 字符或其他会被解析为 UTF-8 的字符，那么可能会被识别并视为 ISO-8859-1 进行处理。

定向格式符

双向控制符可从 编辑 Insert Bidi Control Character 获得。它们可用于：

  • 插入具有强方向性的不可见字符来强制弱方向性或中性方向性字符放在特定位置。

  • 创建一段和所在片段排列方向相反的文本。

这些控制符改变方向但本身不可见。使用 视图 Display Bidi Control Characters 可以在其位置上显示可见的标记。

Mark

要改变具有弱方向性或中性方向性字符（例如标点符号）的位置，请根据片段的方向性在此字符后面插入 LRM 或 RLM 字符：

  • 在从右向左的片段中必须从左向右排列的弱方向性字符（例如阿拉伯语文本中的英语摘录）之后插入 LRM。

  • 在从左向右的片段中必须从右向左排列的弱方向性字符（例如英语文本中的阿拉伯语摘录）之后插入 RLM。

Embedding

Embedding 可用于创建较长的必须与所在片段反向排列的文本段（包含多个单词和空格）。可以根据所在片段的方向性创建两种不同的嵌入文本：

  • 要在从右向左的片段中创建从左向右的嵌入文本，请插入一个 left-to-right embedding（LRE）字符，输入或插入从左向右的文本，然后再插入 pop directional formatting（PDF）字符。

  • 要在从左向右的片段中创建从右向左的嵌入文本，请插入一个 left-to-right embedding（LRE）字符，输入或插入从右向左的文本，然后再插入 pop directional formatting（PDF）字符。

The OmegaT interface generally does not rely on buttons to give access to its functions. Instead, they are called from the menus or, for the majority of functions, from their assigned default shortcut.

Learning the most frequent shortcuts will not take long once you start working with OmegaT. The shortcuts are indicated next to each menu item, allowing them to learn new shortcuts gradually as you use the software.

Customization

OmegaT assigns shortcuts to most of the functions available in the 项目 , 编辑 , and 转到 menus and to a number of functions in the 编辑器 pane. You can also add or modify the shortcuts for most of the functions.

To do so, you have to put the appropriate shortcut definition file in your OmegaT configuration folder. See the 配置文件夹 appendix for details.

There are two shortcut definition files.

MainMenuShortcuts.properties

The shortcut definition file for the menus and a few other items.

EditorShortcuts.properties

The shortcut definition file for the editor.

OmegaT must be restarted after a shortcut definition file has been modified for the new shortcuts to take effect.

Windows and Linux default shortcuts

  • MainMenuShortcuts.properties

  • EditorShortcuts.properties

macOS default shortcuts

  • MainMenuShortcuts.mac.properties

  • EditorShortcuts.mac.properties

The macOS files must be renamed MainMenuShortcuts.properties and EditorShortcuts.properties for OmegaT to recognize them.

The next section describes the syntax used in the shortcut definition files, and provides an example modification.

Syntax

The basic syntax of the shortcut definitions files is simply:

function code=shortcut

Use the tables in the Lists of functions and codes section below to find the values for function code.

The shortcut represents the key combination pressed by the user. It takes the following form:

  • 0 or more modifier

  • followed by 0 or 1 event

  • followed by 1 key

  • where modifier can be: shift, ctrl, meta, alt, or altGraph

    注意

    meta refers to the key with the Windows logo on most keyboards for Windows or Linux systems, and to the command on macOS.

    altGraph refers to the Alt key to the right of the spacebar on keyboards with two Alt keys.

  • event can be: typed, pressed, released

  • and key can be any key available on your keyboard. You can refer to the table presenting the different editor shortcuts to find the values for keys such as Home, Page Up, or the arrow keys.

Empty lines and comments can be added to organize the list and make it easier to read. A comment line starts with a #, and everything after that is ignored by the application.

The easiest way to modify the shortcuts is to download copies of the default files to your configuration folder, as noted above, and make the changes you want there.

例 31. Modifying a shortcut

The default shortcut for closing a project is defined on Windows and Linux as:

projectCloseMenuItem=ctrl shift W

and on macOS as:

projectCloseMenuItem=meta shift W

However, you may want to remove the S key from the shortcut to make it only Ctrl + W (or Command + W on macOS) to match the shortcut you use in other applications.

To do so, modify the MainMenuShortcuts.properties as follows for Windows or Linux:

projectCloseMenuItem=ctrl W

or as follows for macOS:

projectCloseMenuItem=meta W


例 32. Adding a shortcut

If your language pair calls for the frequent use of alternative translations, you may want to assign a shortcut to that function since it does not have one by default.

The steps below demonstrate how to assign the Alt + X shortcut to the 编辑 创建不同译文 menu item.

  1. Open the MainMenuShortcuts.properties you have copied to your configuration folder in a text editor.

  2. As shown in the Edit menu table below, the function code for the Create Alternative Translation function is editMultipleAlternate.

    Searching for that code in the file will bring you to the following line:

    # editMultipleAlternate=

  3. The line is currently a comment. Delete the # at the beginning of the line so OmegaT will recognize the shortcut, and add alt X after the = sign at the end of the line:

    editMultipleAlternate=alt X

  4. Save and close the file. The next time you start OmegaT, your new shortcut should be active and displayed next to the name of the function in the menu.


Save the file after you have finished making your changes. If OmegaT is open, you will have to restart it for your changes to take effect.

Your modified or added shortcuts should now be displayed next to the menu items you have changed. They will now be available in OmegaT as long as there are no conflicts with other functions or with system-wide shortcuts.

应用程序文件夹

应用程序文件夹中包含 OmegaT.jar 应用程序和许多其他重要文件。

注意

应用程序文件夹的位置取决于您的平台和安装 OmegaT 的方式。建议或默认位置如下：

Windows

C:\Program Files\OmegaT\

Linux

/opt/omegat/

macOS

/Applications/OmegaT.app/Contents/Java/

其他平台

/opt/omegat/
